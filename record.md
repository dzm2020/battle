📅 基于 Ark ECS 的战斗库开发计划
以下是完整的开发路线图，按里程碑推进，每个阶段产出可测试的功能模块。

🎯 第一阶段：基础框架搭建（第 1 周）
目标：建立项目骨架和 ECS 核心机制

任务清单
初始化 Go Module，引入 github.com/mlange-42/ark/ecs

设计目录结构（component/system/factory/query/event/config）

定义基础组件（Health、Attributes、Combatant 等纯数据 struct）

实现实体工厂的基础版本（支持单个/批量创建单位）

封装 Ark 的 Mapper 和 Filter（避免业务代码直接依赖 Ark API）

编写单元测试验证组件注册和实体创建

验收标准
能通过配置创建 10000 个战斗实体，耗时 < 50ms

组件内存布局连续，无额外 GC 压力

⚔️ 第二阶段：伤害结算核心（第 2-3 周）
目标：实现完整的伤害-生命值闭环

任务清单
定义伤害相关组件（PendingDamage、DamageType 枚举）

实现 DamageSystem（核心职责：读取 PendingDamage，计算最终伤害）

实现 HealthSystem（核心职责：应用伤害，处理死亡）

设计系统调度顺序（Damage → Health → Death）

添加基础计算公式（物理/魔法/真实伤害、护甲减免）

集成事件系统（DamageEvent、DeathEvent）

压力测试：1000 个单位同时受到伤害的性能

验收标准
单帧处理 10000 次伤害计算 < 1ms

死亡单位正确移除 Combatant 标识

事件能正确触发并被监听

✨ 第三阶段：Buff/Debuff 系统（第 4-5 周）
目标：实现灵活的状态效果系统

任务清单
设计 Buff 组件结构（ID、堆叠层数、持续时间、效果列表）

实现 BuffSystem（持续时间递减、过期自动移除）

添加持续效果机制（DoT 伤害、HoT 治疗）

实现属性修改型 Buff（临时增加攻击力、护甲等）

添加控制效果（眩晕、沉默等，禁用特定行为）

实现 Buff 堆叠逻辑（独立/刷新/合并策略）

提供 Buff 配置接口（数据驱动）

验收标准
单个实体可同时拥有 20+ 个不同 Buff

DoT 效果正确按帧触发伤害

眩晕状态下正确跳过单位行动

🎮 第四阶段：技能系统（第 6-7 周）
目标：构建可配置的技能施放框架

任务清单
定义技能配置结构（消耗、冷却、目标类型、效果列表）

实现 SkillUser 组件（技能列表、法力值、冷却管理）

开发 SkillSystem（处理技能施放请求）

实现冷却时间系统（CooldownSystem）

添加资源消耗系统（法力、怒气、能量等）

实现技能效果与 Buff 系统的联动

支持瞬发技能和吟唱技能

验收标准
技能配置完全由 JSON/YAML 定义，不修改代码

冷却和资源消耗正确扣除

范围技能（如群体伤害）正确影响多个目标

🎲 第五阶段：战斗逻辑完善（第 8-9 周）
目标：补齐战斗常见机制

任务清单
实现暴击/格挡/闪避/命中系统

添加治疗系统（HealSystem，支持直接治疗和 Hot）

实现仇恨/威胁值系统（适用于 RPG 战斗）

添加战斗日志系统（记录每帧所有战斗事件）

实现战斗状态机（战斗开始/结束、回合制/实时切换）

支持多阵营战斗（>2 个敌对阵营）

添加战斗重置和清理机制

验收标准
支持完整的暴击计算链（判定 → 倍率 → 伤害）

仇恨系统正确影响 AI 目标选择

战斗结束后所有临时状态正确清理

✅ **实现摘要（第五阶段）**：详见仓库 [`docs/phase5-battle.md`](/docs/phase5-battle.md)（命中/暴击/治疗/仇恨/日志/多阵营清理与系统顺序）。

🗄️ 第六阶段：数据驱动与配置化（第 10-11 周）
目标：所有战斗内容外部配置

任务清单
设计配置 Schema（单位、技能、Buff 的 JSON 定义）

实现配置加载器（支持热重载）

构建配置验证工具（检查引用完整性、数值合理性）

实现单位模板系统（类似英雄模板 → 实例化）

添加表达式计算支持（技能伤害 = 攻击力 * 1.5 + 固定值）

支持条件效果（如果目标生命 < 30%，则伤害增加 50%）

编写配置示例库（战士、法师、刺客等基础模板）

验收标准
新增一个单位类型只需添加 JSON 文件，无需改代码

热重载后新单位能立即创建

配置错误时给出明确错误信息（如 Buff ID 引用不存在）

⚡ 第七阶段：性能优化（第 12 周）
目标：达到生产级性能

任务清单
使用 pprof 识别性能瓶颈

优化查询：复用 Filter 对象，避免每帧创建

批量操作：使用 NewBatch 替代循环创建

减少组件访问：缓存 Mapper 指针

内存池化：复用事件对象和临时数据结构

添加并行支持：分析 System 间的读写依赖，并行执行无冲突的 System

编写 Benchmark 套件（实体创建、伤害结算、Buff 更新）

性能目标
操作	目标耗时
创建 10000 个实体	< 50ms
单帧更新 10000 个单位	< 2ms
应用 10000 次伤害计算	< 1ms
内存分配	< 100KB/帧

如果决定使用 ECS，一个很好的切入点是从技能/Buff 系统开始，这是 ECS 灵活性的绝佳展示场景。你可以先尝试实现一个 DamageSystem 和一个 BuffSystem，就能很快体会到数据解耦带来的便利。

如果你想进一步了解在 Go 语言中如何具体实现一个战斗循环（例如，如何组织 System 的调度顺序），我可以为你提供更详细的示例
